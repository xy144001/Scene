# TreeSAGE Layout Debugging

These notes are empirical TreeSAGE Flow 2 lessons. Treat them as debugging priors, not universal layout laws. Promote a rule to a hard skill only after it appears repeatedly or is verified by a rendered reference-view comparison.

## When To Use

- A repair makes validators greener but the rendered scene looks worse.
- The reference view is oblique, bright, cropped, or weak for exact global front/back and left/right ordering.
- A local fix moves large anchors, wall branches, repeated objects, or supported children unexpectedly.
- The goal is to preserve an accepted baseline while correcting a small number of visible errors.

## Routing

- Read `references/bedroom-regression.md` for bedroom-specific anchors, wall branches, bedside table pose, doors/windows, and known accepted baselines.
- Read `references/reference-view-depth.md` when global left/right or front/back ordering is unstable, or when DepthAnything/reference-depth evidence is involved.
- Read `references/scale-support.md` for object scale priors, support surfaces, rugs, pillows, tabletop packing, and supported-child drift.
- Read `references/asset-qa.md` for Trellis2/parametric provenance, white bases, visual asset failures, and whether an asset should be accepted, regenerated, or replaced.
- Read `references/asset-generation-prompts.md` only when editing asset prompts, regenerating assets, or debugging Flux/Trellis2 reference-image inputs.

## Core Debugging Procedure

1. Start from the latest accepted rendered baseline when one exists.
2. Change one module or one local correction at a time.
3. Rebuild the GLB, render from the estimated reference camera, and compare against both the reference image and accepted baseline.
4. Inspect reports that correspond to the changed module before accepting the result.
5. Reject candidates that only improve collision, pair constraints, or validator status while making the reference-view composition worse.

## Hard Guardrails

- Collision-free is not an acceptance condition. Final aggregate collision and MuJoCo stability are necessary checks, but the reference-view render remains decisive.
- Preserve branch identity before local wall or pair fixes. Do not split a visually coherent branch to satisfy an isolated wall, ordering, or bbox-edge cue.
- Fix scale before trusting layout repair. Collision and ordering modules solve the wrong problem when major anchors have already been shrunk, enlarged, or misclassified.
- Do not infer asset generator provenance from a directory name. Inspect metadata and the run `asset_source_report`.
- Do not let stale intermediate branch conflicts fail the final result when final aggregate collision is clean; keep stale conflicts as diagnostics.
- Do not pack every side-wall cluster as a front/back wall-depth queue. If the reference group mainly specifies `left_to_right` for a table/storage/seating cluster and only weak or partial `front_to_back`, use a lateral side-wall branch. Use `same_depth_lateral` only when bbox bottom/center-y/depth agree that the members are actually same-depth; otherwise use `mixed_depth_lateral` so the branch keeps lateral order but preserves per-object depth offsets.
- Do not treat `near wall` as `against wall`. `near` may keep an object in the same side-wall branch, but only `against_wall`, `attached_to`, `mounted_on`, `hanging_on`, or `wall_flush` may hard-clamp floor objects to the wall plane or force wall-facing storage pose.
- When a local side-wall branch omits an adjacent bookcase/cabinet that appears in the global floor order, inherit that same-side global anchor into the branch lateral order before packing. A split local/global scene graph should still produce one merged `bookcase -> desk -> chair` side-wall chain.
- When a branch is explicitly classified as `same_depth_lateral` or `mixed_depth_lateral`, do not let older same-wall `left_of/right_of` logic remap that relation to depth. The side-wall branch mode owns the lateral axis semantics, and stale agent spatial-order relations should be softened or suppressed.
- Rebuild final validation reports after the last pose, collision, support, or reference-consistency pass. A final render can be correct while stale `reference_alignment` or `spatial_order` JSON still reflects an earlier bad plan.
- Before final GLB assembly/render, run a supported-child surface settle for tabletop and shelf children. Final anchor motion can leave lamps, plants, books, or decor at an old absolute xy even though their `support_id` is still correct.
- Support settling is not tabletop layout. For nightstands, dressers, desks, and shelves with visible local decor, the reference/tabletop `left_to_right` or `front_to_back` order must feed the tabletop packer and must be rerun after density backoff, fine-stage parent moves, final support snapping, and render-pose feedback.
- For reference bboxes clipped by the image boundary, do not expand storage/bookcase scale to a full semantic default unless the full object is visible. Edge-cropped bookcases/shelves should preserve the normalized scene-graph scale or use a conservative visible-part prior.
- Registered support snapping may preserve edge packing for real bedside/nightstand tabletop children, but not for generic shelf/bookcase children. If a shelf child is too large for the inset support interval, center it on the support surface so support validity is physical.
- Do not use `front_yaw_offset_degrees` as a mesh-local rotation during assembly. It is semantic front-axis metadata used by layout/pose reasoning; physical local mesh normalization must use a separate `asset_local_yaw_offset_degrees` field.
- For back-to-wall beds/sofas, validate the final GLB visual pose, not just `_effective_yaw` and bbox footprint. The scene footprint should keep bed/sofa `width` tangent to the contacted wall and `length/depth` extending into the room; do not rotate the larger numeric dimension to the wall normal. For north/south walls the footprint yaw should be 0 mod 180, and for east/west walls it should be 90 mod 180. Mesh-local correction should align the raw asset wide axis to the wall tangent; use detected headboard/backrest geometry only when it preserves that axis or has high confidence.
- For repeated back-to-wall beds, identical semantic pose does not imply identical visual pose. Compare each Trellis2 asset's canonical front independently; when an asset's raw front differs from the pipeline's normalized bed-local front, write the correction to `asset_local_yaw_offset_degrees` and keep semantic `front_yaw_offset_degrees` normalized. Later wall-pose repair must not erase this mesh-local normalization.
- A nonvisual canonical-front fallback must not overwrite accepted wall-flat poses. If turntable rendering or the canonical-front agent fails and the report falls back to local directional defaults, protect wall art, doors, windows, curtains, mirrors, and posters from new `front_yaw_offset_degrees`; keep the pose-review or wall-flush yaw instead and log the skipped fallback.
- For desk/table plus chair clusters, collision exemption is not placement. A workstation chair must be explicitly translated partly under the table with controlled footprint overlap; side-wall lateral packing should use this tuck geometry instead of leaving ordinary positive clearance between desk and chair.
- For desk/table plus chair clusters, later motion of the table anchor must move and retuck the chair as a functional companion. Supported-child settling does not cover floor chairs, so side-wall reference-depth, density, per-wall, or final-polish moves can otherwise leave the chair at an obsolete absolute coordinate beside the bed or window.
- Generic collision repair must not permanently solve a desk-chair collision by moving the chair away as an ordinary floor branch. After any collision pass that touches a workstation table or chair, rerun the desk-chair tuck solver and validate the bounded tuck collision exemption.
- Desk/table anchor detection must reject supported tabletop children before matching id/category keywords. Object ids like `desk_plant`, `table_lamp`, `books_on_table`, and `photo_frame` are not table anchors just because they contain `desk` or `table`, and must not seed a workstation cluster.
- Desk-chair collision exemptions must be bounded by shallow overlap. A valid tuck may overlap a small front strip of the chair footprint with the table, but full-depth or full-width penetration should remain a collision and trigger repair.
- Do not replace a readable Trellis2/reused door or closet-door asset with parametric fallback only because QA says the wall-attached asset needs a thinner footprint or extreme axis compression. Treat that as a placement/axis issue unless there are real component, fragmentation, white-base, or missing-asset failures.
- Do not replace readable Trellis2 curtain panels with procedural fallback when asset QA is already `ok`. Curtain fallback is for missing, fragmented, blank, unreadable, or explicitly blocking assets; otherwise preserve the generated fabric geometry and let the curtain-specific assembly solver handle symmetry and scale.
- For Trellis2/glTF curtain panels, do not change `asset_axis_to_z` from the imported default using only raw `trimesh` local bounds. The raw mesh axis order can disagree with the Blender-imported visual orientation; keep the default unless a direct Blender six-axis render proves the curtain face is sideways.
- Wall-attached room doors need a floor-grounded panel and a plausible wall-tangent span, not only thin wall depth. After thin-depth clamping, preserve at least a single-door wall-axis width, use a wider bound for double/closet/sliding doors, and debug absolute wall placement separately from same-wall dresser/wardrobe ordering.
- A final visual critic should gate only validated local repairs during an ordinary run. Do not let critic text directly change source code, skills, or experience entries; promote those only from repeated, classified, before/after-improving repairs.
- Treat final-critic object moves, object scales, yaw snaps, bed-depth limits, and wall-art/light placement edits as scene-specific runtime repairs by default. A critic observation becomes a general flow/skill/experience candidate only when it repeats across references or maps to a clear module invariant, survives before/after critic validation without physics/collision regression, and still requires human review before promotion.
- In oblique bedroom references, side-wall storage/table anchors may have bboxes near, but not exactly on, the image edge. Use relaxed left/right wall-affinity arbitration for storage/table objects even when global order is otherwise reliable; keep the relaxation scoped away from beds, rugs, and free decor.
- After room-density compaction or any late pose/layout polish, rerun a per-wall object relation audit. Global left/right or hypergraph membership is not enough for side-wall doors and storage; check wall-local tangent span overlap/gap and repair ordinary room-door/storage conflicts along the wall tangent while preserving wall contact and supported children.
- Before same-wall door/storage auditing, merge duplicated closet-door panels that are adjacent/overlapping parts of one visual closet assembly. The assembly may participate in door/storage gap checks once, but two separately generated full door slabs must not both consume wall tangent/depth.
- Treat against-wall floor furniture as part of the physical floor collision domain. A wardrobe or dresser whose parent/support is `wall_east` can still collide with a bed, nightstand, desk, or chair whose parent is `floor`; validators must check these cross-parent floor contacts, and repairs should prefer moving the free floor/bed visual cluster instead of detaching the wall-backed storage.
- For side-wall storage under a same-wall window, treat the storage/window/curtain stack as a hypothesis, not a hard placement rule. Confirm it with depth/frontness cues and third-stage endpoint-agent relations before anchoring storage to the window segment; thin wall layers may overlap storage along wall-normal thickness when the endpoint relations support that stacking.
- Curtain and drape panels must use curtain-specific axis/scale handling. Do not apply generic wall-flat axis swaps to a curtain panel unless a direct Blender render proves the imported mesh is sideways; preserve paired-panel symmetry and reference vertical extent.
- Window/curtain/rod clusters need an internal vertical invariant, not only left/window/right tangent ordering: paired curtain panels should share height and bottom line, and their top edges should stay close to the window top edge or rod line according to the reference. Use curtain scaling to satisfy this before moving unrelated wall decor or stretching nearby storage.
- If a reference shows a window flanked by two curtain halves but the scene graph has one composite curtain-pair object, split it into separate left/right curtain panel objects before reference grounding and layout. Reuse-only component-policy warnings do not create layoutable instances; the assembly lock needs two panel ids, with the rod kept as a spanning fixture.
- Agent `wall_relation` metadata is not a support-tree edge. Any `attached`, `mounted`, `wall_flush`, or `against` semantic wall relation must be materialized into `attached_to`/`against_wall` relations before support-tree construction, or wall art and panels can silently fall back to floor support and lose their correct height.
- Treat `"wall_id": "null"`/`none` as missing, not as `wall_north`. Wall normalizers and support materializers must require an explicit virtual-wall id, and floor coverings/rugs must skip wall-pose normalization entirely.
- Do not convert `against wall` floor furniture into `attached_to_wall` thin slabs. Beds, sofas, wardrobes, dressers, cabinets, bookcases, nightstands, desks, chairs, floor plants, and floor lamps can touch a wall while remaining floor-supported and keeping full footprint depth; only true wall fixtures such as windows, wall art, mirrors, posters, wall clocks, curtains/rods, and room doors should be clamped to very thin wall-mounted depth.
- Do not classify a composite floor-furniture parent as wall-attached because its description mentions integrated wall details. For example, a bed prompt may say it includes integrated framed wall art or textile details; wall-attached detection should use semantic class, layout role, explicit wall relation, and id/category identity, not free-form description text from asset prompts.
- On east/west side walls, image-space `left_of/right_of` between two objects sharing that wall is only a perspective cue for along-wall tangent depth. Treat both spatial-order-agent and ordinary relation-verifier `left_of/right_of` pairs as soft when they remap to the wall tangent; hard validation/repair should come from wall-local depth, corner affinity, tangent overlap/gap, and room-door/storage separation.
- For storage/dresser/wardrobe assets with a high-confidence canonical front, final pose setters must align the real rendered front (`yaw + footprint_yaw_offset + asset_local_yaw_offset + canonical_front_axis`) to the target. A logical `_effective_yaw` pass is insufficient because Blender does not use `front_yaw_offset_degrees` as a mesh rotation.
- For wall-against floor storage such as wardrobes, dressers, cabinets, chests, sideboards, and bookcases, default to back-to-wall contact with the drawer/door/front face pointing into the room: `wall_east=180`, `wall_west=0`, `wall_north=270`, `wall_south=90`. Do not infer side-face wall contact only because the oblique reference shows handles or drawer fronts; require explicit side-panel/side-face contact evidence before allowing `left_side` or `right_side`.
- Density-stage collision repair must not detach wall-anchored floor anchors from their contacted wall along the wall normal. Beds, sofas, dressers, wardrobes, cabinets, and similar against-wall anchors should keep their wall-contact gap; resolve collisions by lateral/tangent movement, moving bedside/free children, or shrinking/settling the offending side object before pulling the anchor into the room.
- Density-stage foreground/front-depth trimming must not pull east/west wall-backed floor storage along the wall tangent. Wardrobes, dressers, cabinets, chests, bookcases, and sideboards that are against or supported by a side wall should keep their along-wall depth through the shell trim, then be checked by reference-depth tangent alignment and a final per-wall object relation audit. Always rerun that per-wall audit after final density, support, tabletop, and collision repair so corner affinity and room-door/storage tangent separation cannot be overwritten.
- Side-wall storage depth/order evidence must be executed, not only diagnosed. If reference bbox-bottom/frontness or wall-local corner affinity says a side-wall dresser/wardrobe/cabinet should move along the wall tangent, move the storage object itself and translate its direct supported tabletop children with it before support settling and collision validation.
- Fine-stage side-wall edge residuals must gate acceptance. If same-wall wall decor, side-wall storage, and a bed/front anchor imply edge alignment, zero collision is not enough; compare wall-tangent edge spans, scale the storage or bed with the relevant back edge anchored when translation cannot satisfy the relation, reserve or shrink curtain slots around hard window panels, and reject the fine stage if required residuals remain unresolved.
- Fine-stage edge sorting must compare object extremes and fixed clusters, not only centers. For a side-wall window assembly, keep the order `half-curtain -> window -> half-curtain` and compact both curtain halves before extending nearby storage; then check wall decor/storage/bed/door front-edge residuals and grounded bed/table-lamp top-edge residuals as required relations.
- Fine-stage detailed order should come from an agent as endpoint constraints, not from one-off hand-coded scene facts. Ask the agent to compare the reference/render and output x/y/z `low|center|high` relations for layout units, including interval orders such as nested front/back edges and top-edge alignments. The deterministic solver may translate roots or scale grounded height, but a required endpoint relation that remains unsatisfied must block fine-stage acceptance.
- Fine-stage endpoint constraints need a consistency pass before solving. If the evidence says a narrower object is contained inside a wider span, normalize to `wider.low <= narrower.low` and `narrower.high <= wider.high`; do not let reversed structured JSON force the narrow object to contain the wide object.
- In oblique bedroom views, do not let a weak image-space gap between a side-wall desk/storage object and the nearest bedside table push the main bed branch away from the room center. Treat that gap as a one-sided correction for an over-left or over-crowded bed branch; if the bed is already at the center prior, suppress positive rightward shifts and log the suppressed evidence. Repeated reference-consistency passes must not oscillate between a center-bed prior and a storage-gap prior.
- Do not reapply absolute side-wall tangent-depth targets in a residual density pass. Once the first density pass has used reference bbox/depth to correct east/west wall storage, doors, windows, curtains, or wall art, the residual pass should preserve those wall-local tangent coordinates and only trim remaining room shell/foreground space. Repeated bbox-backness corrections are cumulative and can push same-wall objects too far forward or too tightly together even when collision/MuJoCo stay green.
- Treat density-agent shrink ratios as an upper bound. If the full second-stage candidate creates new aggregate collisions or MuJoCo instability, do not immediately discard all density refinement. Run a backoff line search, especially for `front_depth_trim`: first keep width unchanged and apply only foreground/depth trim, then try smaller depth-trim fractions. Each attempt must rerun support settling, tabletop packing, per-wall audit, local collision repair, aggregate collision, and MuJoCo before acceptance; otherwise a small repairable support/tabletop collision can wrongly reject the whole density stage.
- Treat reference depth as a review trigger, not an automatic truth source. DepthAnything/frontness samples and hypergraph depth-profile violations are useful for finding suspicious depth mismatches, but tall storage, windows, curtains, wall art, and oblique crops can produce unreliable absolute depth. Before applying a hard repair from depth alone, compare the reference image, final render, bboxes, and depth map evidence and decide whether the mismatch is a real layout-depth error or a depth-estimation artifact.
- A rug/carpet may be modeled as a thin visual floor layer in the support tree, but it should keep an independent visual material at assembly/render time. Sampling or changing the floor material must not recolor the rug so strongly that the rug disappears into the floor.
